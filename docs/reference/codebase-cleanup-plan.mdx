# Codebase cleanup plan

This page captures the current cleanup plan for `mux` based on a quick repository audit on April 18, 2026.

The goal is to reduce code size and maintenance cost without changing user-visible behavior.

## Objectives

1. Remove dead code and stale compatibility paths that no longer protect real users.
2. Split large multi-responsibility files into smaller modules with clearer ownership.
3. Reduce render churn and event-processing cost in the browser.
4. Shrink lint/test suppression surfaces so real regressions are easier to spot.

## High-signal findings

### 1. Several core files are still monoliths

Current hotspots from a quick line-count scan:

| File                                                       | Approx. LOC | Why it matters                                                                                                       |
| ---------------------------------------------------------- | ----------: | -------------------------------------------------------------------------------------------------------------------- |
| `src/node/services/workspaceService.ts`                    |        7442 | Blends workspace lifecycle, runtime orchestration, archive handling, plan-file logic, and send-message coordination. |
| `src/node/services/agentSession.ts`                        |        5337 | Owns startup recovery, retrying, compaction follow-ups, attachments, and agent switching.                            |
| `src/browser/stores/WorkspaceStore.ts`                     |        4163 | Central browser state path for transcript, usage, activity, and hydration.                                           |
| `src/node/services/providerModelFactory.ts`                |        2095 | Large provider branch tree with custom fetch, routing, auth, and provider-specific policy behavior.                  |
| `src/browser/utils/messages/StreamingMessageAggregator.ts` |        3237 | Hot path for transcript/event merging and likely performance-sensitive.                                              |
| `src/browser/components/ProjectSidebar/ProjectSidebar.tsx` |        3065 | Large UI surface with multiple behaviors and memoization pressure.                                                   |
| `src/browser/features/ChatInput/index.tsx`                 |        2840 | Large interaction surface with editing, attachments, submit rules, and tool affordances.                             |

### 2. Legacy compatibility branches need a focused audit

Potential dead-code candidates are not random exports; they are compatibility paths that may no longer be needed:

- `src/node/services/agentSession.ts:143-159` — legacy `continueMessage` support inside compaction request metadata.
- `src/node/services/agentSession.ts:4808-4851` — legacy `mode`/fallback agent resolution for persisted follow-ups.
- `src/node/services/workspaceService.ts:1982-2039` — dual new/legacy plan-file path handling.
- `src/node/services/workspaceService.ts:6106-6169` — best-effort deletion of both new and legacy plan paths.
- `src/node/services/workspaceService.ts:488-498` — legacy fork naming support.

These branches may still be required, but they should be proven with current fixtures and migration expectations rather than left in place indefinitely.

### 3. Lint suppressions are concentrated in a few files

The repository contains a large number of `eslint-disable` suppressions. The highest-value audit targets are:

- `src/browser/stores/WorkspaceStore.test.ts`
- `src/node/services/workspaceService.test.ts`
- `src/browser/components/ProjectSidebar/ProjectSidebar.tsx`
- `src/browser/features/ChatInput/index.tsx`
- `src/browser/features/RightSidebar/RightSidebar.tsx`
- `src/browser/utils/highlighting/highlightWorkerClient.ts`

The tests likely contain many acceptable suppressions, but they should still be clustered and justified. Production-file suppressions should be reduced first.

### 4. There are clear performance-sensitive browser paths

Likely optimization candidates:

- `src/browser/utils/messages/StreamingMessageAggregator.ts` — event merge path, transcript truncation, TODO/status updates.
- `src/browser/stores/WorkspaceStore.ts` — broad state ownership risks expensive fan-out updates.
- `src/browser/components/ProjectSidebar/ProjectSidebar.tsx` — large render surface tied to frequently changing state.
- `src/browser/features/ChatInput/index.tsx` — complex interaction tree likely to re-render more than necessary.

### 5. Provider logic is still branch-heavy

`src/node/services/providerModelFactory.ts` still contains a long provider-specific branch ladder. Even after recent Ollama cleanup, the file is carrying:

- route resolution
- auth resolution
- fetch wrapping
- provider-specific option normalization
- gateway behavior
- model construction

This is a good candidate for adapter extraction without changing behavior.

## Recommended workstreams

### Workstream 1: Dead-code and compatibility audit

### Scope

- Audit legacy branches in `workspaceService.ts` and `agentSession.ts`.
- Confirm which persisted formats are still produced by current code.
- Remove dead compatibility code only after tests lock the migration behavior.

### Deliverables

- A short inventory of every legacy path and the fixture/test that still exercises it.
- Removal of compatibility paths that have no current producer and no required migration coverage.
- Regression tests for any compatibility path that survives.

### Success criteria

- No unsupported legacy branch remains without a corresponding test or migration note.
- Reduced branch count in `workspaceService.ts` and `agentSession.ts`.

### Workstream 2: Monolith decomposition

### Scope

Split by responsibility, not by arbitrary size targets:

- `workspaceService.ts`
  - workspace lifecycle
  - archive/unarchive
  - runtime selection/bootstrap
  - plan-file operations
  - heartbeat/settings helpers
- `agentSession.ts`
  - startup recovery
  - compaction follow-up parsing
  - retry/autoretry state
  - attachment/file-change tracking
  - agent handoff/switching
- `providerModelFactory.ts`
  - provider adapter modules
  - shared fetch wrappers
  - shared credential gates

### Deliverables

- Smaller focused modules with no behavior change.
- Tests moved with the extracted modules where practical.

### Success criteria

- Reduced file size and import surface in the top-level orchestrators.
- Fewer unrelated responsibilities per file.

### Workstream 3: Browser-state and rendering optimization

### Scope

- Profile and reduce update fan-out from `WorkspaceStore`.
- Audit `StreamingMessageAggregator` for repeated scans, cloning, and broad invalidation.
- Push high-frequency subscriptions down to leaf components where possible.

### Candidate targets

- `src/browser/stores/WorkspaceStore.ts`
- `src/browser/utils/messages/StreamingMessageAggregator.ts`
- `src/browser/components/ProjectSidebar/ProjectSidebar.tsx`
- `src/browser/features/ChatInput/index.tsx`

### Success criteria

- Fewer workspace-wide re-renders during streaming.
- Less work per transcript event.
- No new `useMemo` / `useCallback` cargo-culting; prefer state-shape cleanup and subscription colocation.

### Workstream 4: Suppression and test cleanup

### Scope

- Remove stale `eslint-disable` lines in production files first.
- Convert broad test suppressions into typed helpers or local fixtures.
- Replace `any`-heavy test setup with typed builders where repetition is high.

### Candidate clusters

- `src/browser/stores/WorkspaceStore.test.ts`
- `src/node/services/workspaceService.test.ts`
- `src/browser/utils/messages/*.test.ts`

### Success criteria

- Smaller suppression footprint.
- Better signal from lint and type failures.

### Workstream 5: Generated and embedded content review

### Scope

- Audit `src/node/services/agentSkills/builtInSkillContent.generated.ts` for load-time and review-cost impact.
- Confirm whether the entire generated blob must load eagerly.

### Questions to answer

1. Can built-in skill content be loaded on demand?
2. Can generation be chunked by skill family?
3. Can tests assert behavior without materializing the whole blob in every path?

### Success criteria

- Lower memory/startup pressure or, at minimum, a documented reason for keeping the eager model.

## Suggested execution order

1. **Compatibility audit** — safest path to real dead-code removal.
2. **Provider factory extraction** — smaller blast radius than the workspace/session services.
3. **Browser hot-path profiling** — optimize streaming and sidebar churn.
4. **Workspace/session decomposition** — highest payoff, highest coordination cost.
5. **Suppression cleanup** — easier after structural cleanup reduces noise.

## Validation gates

Every cleanup slice should keep the same verification bar:

- `make typecheck`
- targeted `bun test ...`
- `make lint`
- targeted UI regression checks for browser-state changes

For dead-code removal specifically:

- add or update tests before deleting compatibility paths
- verify no remaining callers or persisted producers still rely on the removed path

## First tracer-bullet tasks

If starting immediately, begin with these small slices:

1. Extract Ollama/OpenAI/OpenRouter adapter helpers from `providerModelFactory.ts`.
2. Audit legacy plan-path handling in `workspaceService.ts`.
3. Audit legacy compaction follow-up parsing in `agentSession.ts`.
4. Reduce one production-file suppression cluster in `ProjectSidebar.tsx` or `ChatInput/index.tsx`.
5. Measure `WorkspaceStore` + `StreamingMessageAggregator` update frequency during one long streaming session.

## Out of scope for this pass

- Rewriting generated content pipelines without a before/after measurement.
- Cosmetic churn that only changes naming or formatting.
- New dependencies just for dead-code discovery; prefer existing compiler/lint/test signals first.
